# Contributing

## Before opening a PR

- Open an issue first to discuss the change, unless it's a clear bug fix or typo.
- One skill (or one focused change) per PR. Keep scope narrow.

## Guidelines

- Match the existing style and voice in skill files — terse, directive, no filler.
- Rigid skills must stay rigid. Don't soften language or add escape hatches.
- If you're adding a skill, check whether it belongs as a section in an existing one first.
- Do not add `docs/jds/` files to commits — these are JDS working artifacts and are gitignored.

## Skill types

**Rigid** skills define non-negotiable protocols (TDD, debugging, verification). Follow them exactly; don't adapt tone or add exceptions.

**Flexible** skills define guiding principles that adapt to context. Maintain the structure; allow the content to flex.

Each skill declares its own type — match it.
